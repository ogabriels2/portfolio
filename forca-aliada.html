01 // visão geral

O que é a Força Aliada?

A Força Aliada é uma comunidade Minecraft Survival Vanilla sem resets, sem pay-to-win, que existia desde 2021 como um servidor privado entre amigos — rodando via Hamachi, sem nenhuma pretensão além de jogar. Em 2024 ganhou identidade visual, em 2025 foi ao ar com domínio próprio e suporte a Bedrock, e em 2026 o projeto escalou para um ecossistema digital completo.

O projeto vai muito além de um site institucional. É uma plataforma social completa: autenticação com PIN de e-mail e Microsoft/Xbox OAuth, feed de posts com mídia moderada por IA, sistema de follows, enquetes com expiração, ranks gamificados, sistema de Mérito e Capital, e app desktop Windows com console ao vivo e auto-updater.

Cada decisão técnica resolveu um problema real do mundo: detecção de proxy TCPShield com 4 heurísticas em cascata, moderação de imagens em 3 camadas via Google Vision, procedures e triggers PostgreSQL para personalização de feed, e IPC com whitelist de canais no Electron para sandboxing seguro.

Moderação de imagens por IA — Google Vision SafeSearch em 3 camadas

Antes de qualquer upload para o Cloudinary, o arquivo passa pela Google Vision SafeSearch API. O sistema classifica em 3 camadas: aprovado (sobe normalmente), suspeito (sobe, mas entra na fila de revisão do admin) e rejeitado (bloqueado imediatamente com log de auditoria). Fail-open intencional: se a API estiver indisponível, o upload prossegue sem travar o usuário. A lógica de 3 camadas evita tanto falsos positivos quanto conteúdo claramente impróprio.

02 / 03
🔮

Máquina de estados do servidor Minecraft — 4 heurísticas anti-TCPShield

O TCPShield (proxy anti-DDoS) responde ao ping de status mesmo quando o servidor real está offline — quebrando a detecção ingênua. A solução usa uma máquina de 3 estados (unknown/offline/online) com 4 heurísticas em cascata: (1) UDP Query porta 25565 — TCPShield não proxeia UDP; (2) comparação do MOTD de offline configurado; (3) consulta cross-datacenter via mcstatus.io; (4) confirmação por contagem de players. O sistema nunca reporta falso "online" após shutdown explícito.

javascript src/server.mjs 
// Máquina de 4 heurísticas — TCPShield responde "online" mesmo offline
async function resolveServerState(host) {
  // Heurística 1: UDP Query — TCPShield não proxeia UDP
  const udp = await queryUDP(host, 25565).catch(() => null);
  if (!udp?.players) return setState('offline'); // offline real

  // Heurística 2: MOTD offline configurado no proxy
  const ping = await pingServer(host);
  if (ping.description === process.env.OFFLINE_MOTD)
    return setState('offline');

  // Heurística 3: Cross-datacenter via mcstatus.io
  const { online } = await mcstatus.fetch(host);
  if (!online) return setState('offline');

  // Heurística 4: Confirmação positiva por player count
  return setState(ping.players.online > 0 ? 'online' : 'unknown');
}

Afinidade de tags com PostgreSQL — procedures e triggers nativos

A rede social usa procedures e triggers nativos do PostgreSQL para manter um score de afinidade por hashtag por usuário. Curtidas (+1), comentários (+2) e reposts (+3) disparam a função fa_apply_tag_affinity() via trigger, e o score é clampeado entre 0 e 10. Toda a lógica vive no banco — zero processamento externo, zero latência adicional.

sql migrations/tag_affinity.sql 
-- Procedure chamada por triggers em likes, reposts e comments
CREATE OR REPLACE FUNCTION fa_apply_tag_affinity()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE
  v_weight INT := CASE TG_TABLE_NAME
    WHEN 'post_likes'    THEN 1  -- curtida: +1
    WHEN 'post_comments' THEN 2  -- comentário: +2
    WHEN 'post_reposts'  THEN 3  -- repost: +3
  END;
BEGIN
  INSERT INTO user_tag_affinity (user_id, tag, score)
    SELECT NEW.user_id, unnest(p.tags), v_weight
      FROM user_posts p WHERE p.id = NEW.post_id
  ON CONFLICT (user_id, tag) DO UPDATE
    SET score = LEAST(10,
      user_tag_affinity.score + EXCLUDED.score); -- clamp 0-10
  RETURN NEW;
END;
$$;

Revogação de sessão via hash SHA-256

JWTs são stateless por natureza — mas sessões precisam ser revogáveis individualmente. A solução: ao logar, armazeno o hash SHA-256 do token na tabela user_sessions. Em cada request, re-hasho o token e busco no banco. DELETE na sessão = revogação instantânea, sem blacklist distribuída.

javascript src/server.mjs — middleware requireAuth() 
async function requireAuth(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'Unauthorized' });

  // Hash SHA-256 do token → nunca armazenamos JWT em texto puro
  const tokenHash = crypto
    .createHash('sha256')
    .update(token)
    .digest('hex');

  const { rows } = await pool.query(
    `SELECT u.id, u.username, u.role, s.revoked
     FROM users u
     JOIN user_sessions s ON s.user_id = u.id
     WHERE s.token_hash = $1`,
    [tokenHash]
  );

  if (!rows[0] || rows[0].revoked)
    return res.status(401).json({ error: 'Session revoked' });

  req.user = rows[0];
  next(); // ← JWT válido, sessão ativa
}
04 // detalhes avançados

Funcionalidades que fazem a diferença

Features de produto que revelam maturidade de sistema — pensadas para escala, segurança e experiência real.

Cloud Lockout após shutdown

Quando o app sinaliza serverStopped, os próximos heartbeats ficam bloqueados por 60 segundos. Isso evita que o app reconecte imediatamente e desfaça o modo "cloud detection", dando tempo real para o servidor Java finalizar. O lockout é levantado automaticamente ao expirar ou quando serverStarted:true chega.

Multi-conta Microsoft/Xbox

A tabela user_integrations suporta múltiplas contas Microsoft por usuário. Suporta tanto Java UUID quanto Bedrock XUID sintético ('bedrock_' + xuid), com is_primary marcado e rate-limiting de 3 sincronizações/5min para proteger o App ID da Microsoft contra abuso.

Afiliação de Contas — Escopo por Permissão

A tabela account_affiliations permite que um usuário delegue escopo post a outro. Um delegate pode postar em nome de um owner sem ter acesso total à conta — controle granular de permissão por escopo, com last_used_at rastreado para auditoria.

Capital & Mérito com histórico completo

Cada jogador tem saldo de Mérito (⭐) e Capital (💰) em player_balances, com histórico granular em merit_records e capital_records. Admins podem atribuir, subtrair e visualizar todo o histórico com categoria, razão, responsável e timestamp.

07 // rede social

Rede Social Embutida

O site institucional é também uma plataforma social completa para os membros da comunidade.

Posts com mídia e enquetes

Membros podem criar posts com texto, até 4 imagens (validadas por Google Vision), e enquetes com múltiplas opções e data de expiração. Posts podem ser editados, com edit_count rastreado e histórico preservado.

Curtidas, Reposts e Comentários

Sistema completo de engajamento: curtidas com índice único (post_id, user_id), reposts com dedup automático via índice parcial, e comentários com soft-delete (is_deleted) para preservar threads mesmo com remoção.

Perfis públicos customizáveis

Bio (160 chars), display name, avatar e cover photo, configurações de privacidade por campo. Links externos em JSONB. Visibilidade controlada por public_profile e show_online em user_preferences.

Compartilhamento público com SEO

Todo post tem URL pública com meta tags OG completas, canonical link e schema.org. Compartilhamento nativo via WhatsApp e clipboard. Visualização pública sem login — SEO-friendly por padrão.

Chat integrado — inspirado no WhatsApp + Discord

O site conta com um chat em tempo real com design dark premium inspirado em WhatsApp, Telegram e Discord. FAB launcher com animação spring, bolhas de mensagem com timestamps, tema claro/escuro sincronizado com a preferência salva na API, e tokens CSS customizados (--fc-gold, --fc-spring).
