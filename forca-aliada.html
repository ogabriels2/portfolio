O que é a Força Aliada?

Sob o Capô

Decisões técnicas avançadas que separam projetos amadores de plataformas escaláveis de produção.

Google Vision (SafeSearch) em 3 Camadas

O upload de mídia não é cego. Imagens passam pela IA do Google. São classificadas como Aprovadas (liberadas), Suspeitas (fila admin) ou Rejeitadas (bloqueio imediato + log). Possui fail-open estrutural.

Máquina de Estados Anti-TCPShield

Proxies anti-DDoS mentem o status do servidor. A API usa 4 heurísticas em cascata (UDP Query, MOTD offline, mcstatus.io externo e contagem de players) para calcular o real status do container Java na nuvem.

Procedures e Triggers Nativos

A afinidade de hashtags do feed social é calculada direto no banco via Triggers PostgreSQL. Curtidas (+1), comentários (+2) disparam a procedure fa_apply_tag_affinity() economizando latência do Node.js.

05 // Segurança

Revogação de Sessões via SHA-256

JWTs são stateless, mas sistemas robustos precisam de revogação imediata (logout remoto). A solução implementada faz o hash do token em tempo real e compara com o banco.

Nunca armazenamos o JWT exposto no banco. Um simples DELETE na sessão do banco encerra o acesso no front-end milissegundos depois.

src/middleware/auth.mjs
async function requireAuth(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'Unauthorized' });

  // Hash SHA-256 (Nunca armazenar texto puro)
  const tokenHash = crypto
    .createHash('sha256')
    .update(token)
    .digest('hex');

  const { rows } = await pool.query(
    `SELECT u.id, s.revoked
     FROM users u
     JOIN user_sessions s ON s.user_id = u.id
     WHERE s.token_hash = $1`,
    [tokenHash]
  );

  if (!rows[0] || rows[0].revoked)
    return res.status(401).json({ error: 'Session revoked' });

  req.user = rows[0];
  next();
}

Engenharia // Moderação por IA

Google Vision SafeSearch em 3 Camadas

Cada imagem enviada pela rede social passa por uma pipeline de moderação automatizada antes de ser publicada. A IA do Google classifica adulto, violência, sugestivo e médico em 5 níveis de probabilidade.

Pipeline de Moderação

✅
Camada 1 — AprovadoUNLIKELY / VERY_UNLIKELY / POSSIBLE em todas as categorias. Imagem publicada imediatamente no Cloudinary.
⚠️
Camada 2 — SuspeitoPOSSIBLE (score 3) em adult, violence ou racy. Imagem é publicada mas entra na fila de moderação humana do admin.
🚫
Camada 3 — RejeitadoLIKELY ou VERY_LIKELY (score ≥ 4) em adult ou violence. Upload bloqueado antes do Cloudinary. Log imutável em audit_logs.
🔓
Fail-open IntencionalSe GOOGLE_VISION_API_KEY ausente ou timeout (>10s), a imagem passa sem moderação. Plataforma nunca fica travada por indisponibilidade da API externa.

Categorias Analisadas

adult violence racy medical 5 níveis: VERY_UNLIKELY → VERY_LIKELY

Moderação em fila, não em bloco. O design de "suspeito → fila humana" é intencional: falsos positivos não prejudicam o criador de conteúdo, mas conteúdo realmente duvidoso recebe revisão manual. O admin tem visão clara de tudo pendente no dashboard.

Engenharia // Detecção de Proxy

Máquina de Estados Anti-TCPShield

O TCPShield é um proxy anti-DDoS que intercepta pings de status e devolve dados falsos ("servidor online") mesmo quando o backend real está offline. Quatro heurísticas em cascata resolvem isso definitivamente.

As 4 Heurísticas em Cascata

📡
Heurística 1 — UDP QueryO TCPShield não proxeia UDP. Se queryFull() responder na porta 25565, é confirmação positiva de servidor real. Requer enable-query=true no server.properties.
📝
Heurística 2 — MOTD ComparisonCompara o MOTD retornado com o MOTD offline configurado (MC_OFFLINE_MOTD env var). Match exato = TCPShield respondendo = servidor offline.
🌍
Heurística 3 — mcstatus.io ExternoConsulta de datacenter externo diferente. Se ambos veem 0 players com mesmo MOTD, TCPShield provavelmente está em ambos. Possui cache próprio de ~30s.
👥
Heurística 4 — Players Online > 0Prova positiva definitiva. O Offline MOTD do TCPShield sempre retorna 0 players. Se a contagem for > 0, o servidor definitivamente está rodando.

Máquina de Estados

❓
Estado: unknownEstado inicial. Usa detecção de ping com filtro TCPShield para determinar status real.
🔴
Estado: offlineApp sinalizou shutdown ou ping confirmou offline real. Após 3min sem App conectado, volta para unknown automaticamente.
🟢
Estado: onlineApp sinalizou serverStarted ou ping real respondeu. Invalida cache de status imediatamente na transição.
⏱️
Cache TTL 50sLigeiramente menor que o ciclo de polling do dashboard (60s). Garante dados frescos no próximo poll.

Problema específico, solução específica. Esse sistema existe porque o TCPShield evoluiu com a feature "Offline MOTD" que quebra a detecção por latência (<2ms). A abordagem em camadas garante que mesmo com cache de datacenter externo, o status sempre reflete a realidade.

Engenharia // PostgreSQL Avançado

Procedures e Triggers Nativos

A personalização do feed social não acontece no Node.js — ela acontece dentro do banco de dados. Triggers em PL/pgSQL disparam automaticamente a cada interação e mantêm scores de afinidade por hashtag em tempo real.

Sistema de Afinidade

❤️
Trigger: post_likesINSERT → +1.0 de afinidade nas tags do post. DELETE (unlike) → -1.0. Nunca vai abaixo de 0.
💬
Trigger: post_commentsINSERT de comentário → +2.0 (sinal mais forte que like). Soft-delete → -2.0. UPDATE de is_deleted monitora ambas direções.
🏷️
Extração de HashtagsA function fa_apply_tag_affinity() usa regexp_matches(content, '#([[:alnum:]_]{2,32})', 'g') para extrair tags. Processamento 100% dentro do PostgreSQL.
📊
Score 0 a 10 por Taguser_tag_affinity com score NUMERIC(5,2) constraint CHECK ≥ 0 AND ≤ 10. UPSERT via ON CONFLICT DO UPDATE com LEAST/GREATEST.

Schema Envolvido

user_tag_affinity trg_post_likes_affinity trg_post_comments_affinity fa_apply_tag_affinity() fa_post_likes_affinity_trigger() fa_post_comments_affinity_trigger() PL/pgSQL regexp_matches

Por que no banco, não no Node.js? Triggers executam dentro da transação da interação original — atomicamente. Não há risco de inconsistência por crash do servidor Node entre o INSERT do like e a atualização do score. Zero latência adicional de rede.
