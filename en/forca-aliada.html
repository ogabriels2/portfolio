What is Força Aliada?

Força Aliada started in 2021 as a private Minecraft server among friends, running via Hamachi. In 2025 it opened to the public, and in 2026 it scaled into a cutting-edge digital ecosystem with social networking, messaging, and an admin panel.

The project solves real engineering challenges: TCPShield proxy detection with 4 cascading heuristics, multi-layer image moderation using Google Vision, secure IPC communication in Electron, and 40 relational tables with 138+ REST API endpoints.

04 // Engineering

Under the Hood

Advanced technical decisions that separate amateur projects from scalable production platforms.

Google Vision (SafeSearch) in 3 Layers

Media uploads aren't blind. Images go through Google's AI. They're classified as Approved (released), Suspicious (admin queue), or Rejected (immediate block + log). Has structural fail-open.

Anti-TCPShield State Machine

Anti-DDoS proxies lie about server status. The API uses 4 cascading heuristics (UDP Query, offline MOTD, external mcstatus.io and player count) to compute the real status of the Java container in the cloud.

Native Procedures & Triggers

The social feed's hashtag affinity is calculated directly in the database via PostgreSQL Triggers. Likes (+1), comments (+2) fire the fa_apply_tag_affinity() procedure, saving Node.js latency.

05 // Security

Session Revocation via SHA-256

JWTs are stateless, but robust systems need immediate revocation (remote logout). The implemented solution hashes the token in real-time and compares it to the database.

We never store the exposed JWT in the database. A simple DELETE on the database session ends access on the front-end milliseconds later.

src/middleware/auth.mjs
async function requireAuth(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'Unauthorized' });

  // SHA-256 Hash (Never store plaintext)
  const tokenHash = crypto
    .createHash('sha256')
    .update(token)
    .digest('hex');

  const { rows } = await pool.query(
    `SELECT u.id, s.revoked
     FROM users u
     JOIN user_sessions s ON s.user_id = u.id
     WHERE s.token_hash = $1`,
    [tokenHash]
  );

  if (!rows[0] || rows[0].revoked)
    return res.status(401).json({ error: 'Session revoked' });

  req.user = rows[0];
  next();
}

Engineering // AI Moderation

Google Vision SafeSearch in 3 Layers

Every image uploaded to the social network goes through an automated moderation pipeline before being published. Google's AI classifies adult, violence, suggestive and medical content across 5 probability levels.

Moderation Pipeline

✅
Layer 1 — ApprovedUNLIKELY / VERY_UNLIKELY / POSSIBLE across all categories. Image published immediately to Cloudinary.
⚠️
Layer 2 — SuspiciousPOSSIBLE (score 3) in adult, violence or racy. Image is published but enters the admin's human moderation queue.
🚫
Layer 3 — RejectedLIKELY or VERY_LIKELY (score ≥ 4) in adult or violence. Upload blocked before Cloudinary. Immutable log in audit_logs.
🔓
Intentional Fail-OpenIf GOOGLE_VISION_API_KEY is missing or timeout (>10s), the image passes without moderation. Platform never gets locked by external API unavailability.

Analyzed Categories

adult violence racy medical 5 levels: VERY_UNLIKELY → VERY_LIKELY

Queue moderation, not block moderation. The "suspicious → human queue" design is intentional: false positives don't hurt the content creator, but genuinely questionable content gets manual review. The admin has clear visibility of everything pending in the dashboard.

Engineering // Proxy Detection

Anti-TCPShield State Machine

TCPShield is an anti-DDoS proxy that intercepts status pings and returns false data ("server online") even when the real backend is offline. Four cascading heuristics solve this definitively.

The 4 Cascading Heuristics

📡
Heuristic 1 — UDP QueryTCPShield doesn't proxy UDP. If queryFull() responds on port 25565, it's a positive confirmation of a real server. Requires enable-query=true in server.properties.
📝
Heuristic 2 — MOTD ComparisonCompares returned MOTD against the configured offline MOTD (MC_OFFLINE_MOTD env var). Exact match = TCPShield responding = server offline.
🌍
Heuristic 3 — External mcstatus.ioQuery from a different external datacenter. If both see 0 players with same MOTD, TCPShield is likely on both. Has its own ~30s cache.
👥
Heuristic 4 — Players Online > 0Definitive positive proof. TCPShield's Offline MOTD always returns 0 players. If count is > 0, the server is definitely running.

State Machine

❓
State: unknownInitial state. Uses ping detection with TCPShield filter to determine real status.
🔴
State: offlineApp signaled shutdown or ping confirmed real offline. After 3min without App connected, reverts to unknown automatically.
🟢
State: onlineApp signaled serverStarted or real ping responded. Immediately invalidates status cache on transition.
⏱️
Cache TTL 50sSlightly less than the dashboard polling cycle (60s). Ensures fresh data on the next poll.

Specific problem, specific solution. This system exists because TCPShield evolved with an "Offline MOTD" feature that breaks latency-based detection (<2ms). The layered approach ensures that even with external datacenter cache, status always reflects reality.

Engineering // Advanced PostgreSQL

Native Procedures & Triggers

The social feed's personalization doesn't happen in Node.js — it happens inside the database. PL/pgSQL triggers fire automatically on every interaction and maintain hashtag affinity scores in real time.

Affinity System

❤️
Trigger: post_likesINSERT → +1.0 affinity in post tags. DELETE (unlike) → -1.0. Never drops below 0.
💬
Trigger: post_commentsComment INSERT → +2.0 (stronger signal than like). Soft-delete → -2.0. UPDATE of is_deleted monitors both directions.
🏷️
Hashtag Extractionfa_apply_tag_affinity() uses regexp_matches(content, '#([[:alnum:]_]{2,32})', 'g') to extract tags. 100% processing inside PostgreSQL.
📊
Score 0 to 10 per Taguser_tag_affinity with NUMERIC(5,2) score, CHECK constraint ≥ 0 AND ≤ 10. UPSERT via ON CONFLICT DO UPDATE with LEAST/GREATEST.

Involved Schema

user_tag_affinity trg_post_likes_affinity trg_post_comments_affinity fa_apply_tag_affinity() fa_post_likes_affinity_trigger() fa_post_comments_affinity_trigger() PL/pgSQL regexp_matches

Why in the database, not Node.js? Triggers execute inside the original interaction's transaction — atomically. There's no risk of inconsistency from a Node server crash between the like INSERT and the score update. Zero additional network latency.
